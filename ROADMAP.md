# CRM-Forsage — Production Roadmap

> Проєктний план доробок. Складено архітектором/PM на основі аудиту існуючої системи.
> Дата складання: 2026-05-21.

## Як користуватися цим документом

1. **Один етап = один прогон моделі.** Не змішувати, не виходити за межі «НЕ менять».
2. **Не починати ЭТАП N+1 до підтвердженого «зеленого» ЭТАП N.** «Зелений» = міграція застосована, `npx tsc --noEmit` чистий, smoke-test з блоку «Тест проверки» пройдений на staging БД.
3. **Кожен прогон починається з фіксованого промпта** — див. блок [«Шаблон запуску прогону»](#шаблон-запуску-прогону) внизу.
4. Якщо обсяг етапу не вкладається в одну сесію — **розщепити перед запуском** (5.a / 5.b), не «дописати в наступному».
5. Будь-який вихід за зону «Изменить:» у тому ж прогоні = негативний результат, навіть якщо тести проходять.

# ЭТАП 1 — Доступна кількість як єдине джерело правди (P0)

**Цель:** дати системі чесну метрику `available_qty = qty_on_hand − активні резерви`. Зараз всі модулі дивляться на `qty_on_hand` і брешуть один одному.

**Причина порядка:** без цього шару всі подальші резервування / атомарні видачі будуть інкрементальними милицями. Це фундамент.

**Зависимости:** таблиця `inventory_reserves` (вже існує, міграція 003).

**Использовать существующее:**
- таблицы: `products`, `inventory_reserves`
- сервисы: `productService.ts`
- страницы: `ProductsPage.tsx`, `ProductDetailPage.tsx`, `SearchPanel.tsx` (POS)

**Создать:**
- БД: `CREATE VIEW v_product_stock` з полями `product_id, qty_on_hand, qty_reserved, qty_available`.
- БД: `CREATE FUNCTION get_available_qty(product_id UUID) RETURNS NUMERIC` (для inline use в інших RPC).
- ендпоінт: `GET /api/v1/products/:id/stock` → `{ on_hand, reserved, available }`.

**Изменить:**
- `server/src/services/productService.ts` — додати `getStockBreakdown(id)`.
- `apps/web/src/features/products/productApi.ts` — додати тип `StockBreakdown` і метод.
- `apps/web/src/features/products/ProductDetailPage.tsx` — показати під залишком "у резерві: X / доступно: Y".

**НЕ менять:**
- `process_sale` (буде в Етапі 3).
- POS-логіку відображення кількості (Етап 3).
- Будь-яку логіку customer_orders (Етап 2-4).

**Поток работы:**
1. Міграція: створити VIEW + FUNCTION. View — простий left join з агрегатом `SUM(qty) FILTER (WHERE status='active')`.
2. Endpoint + service.
3. UI на сторінці товару — рядок з трьома числами.
4. Smoke-тест: вручну вставити запис в `inventory_reserves` → переглянути картку → побачити "у резерві".

**Критерий готовности:**
- `SELECT * FROM v_product_stock WHERE product_id='…'` повертає 3 поля для будь-якого товару.
- Сторінка деталей товару показує всі три числа.
- TS-перевірка зелена.

**Тест проверки:**
- Товар без жодного резерву: `qty_reserved=0`, `qty_available=qty_on_hand`.
- Товар з активним резервом 3 шт: `qty_available` менший на 3.
- Скасований резерв (status='canceled') не зменшує `qty_available`.

**Риск:** низький — read-only шар, нічого не ламає.
**Откат:** `DROP VIEW v_product_stock; DROP FUNCTION get_available_qty;` — зворотно за 1 запит.
**Оценка сложности:** S
**Оценка количества прогонов:** 1
**Решения этапа:** менеджер бачить чесну "доступну для продажу" кількість, кашир ще не використовує.

# ЭТАП 2 — Резервування warehouse-позицій у customer_orders (P0)

**Цель:** при додаванні позиції замовлення з `source_type='warehouse'` фізично резервувати товар. При зміні статусу позиції резерв атомарно змінює стан.

**Причина порядка:** без цього Етап 1 — декорація. Має бути ДО Етапу 3, бо `process_sale` має бачити вже існуючі резерви.

**Зависимости:** Етап 1 (VIEW `v_product_stock`).

**Использовать существующее:**
- таблицы: `customer_orders`, `customer_order_items`, `inventory_reserves`
- сервисы: відсутні методи резервації — створюємо
- роуты: `routes/customerOrders.ts` (тільки сегмент `POST /` та `PATCH /:id/items/:itemId/status`, `POST /:id/cancel`)

**Создать:**
- БД: міграція додає `customer_order_items.reserve_id UUID NULL REFERENCES inventory_reserves(id) ON DELETE SET NULL`.
- БД: RPC `reserve_order_item(p_order_id, p_item_id, p_product_id, p_qty)` — створює запис у `inventory_reserves` зі статусом 'active', повертає `reserve_id`.
- БД: RPC `release_order_reserve(p_item_id)` — `UPDATE inventory_reserves SET status='canceled'`.
- БД: RPC `consume_order_reserve(p_item_id)` — `UPDATE inventory_reserves SET status='consumed'`.
- сервіс: `server/src/services/orderReserveService.ts` — три тонких обгортки над RPC.

**Изменить:**
- `server/src/routes/customerOrders.ts`:
  - `POST /` — після створення items, для кожного з `source_type='warehouse' AND product_id IS NOT NULL` → `reserve_order_item`. Якщо резерв не вдався (недостатньо available) — повернути 422 і відкатити insert.
  - `PATCH /:id/items/:itemId/status` — при переході в `canceled` → `release_order_reserve`.
  - `POST /:id/cancel` — release для всіх warehouse-items.
- Не вносити змін у `customer_order_items.source_type='supplier'` flow.

**НЕ менять:**
- `process_sale` (Етап 3).
- UI замовлень — резерв працює прозоро.
- Логіку POS.

**Поток работы:**
1. Міграція: колонка `reserve_id` + 3 RPC.
2. Сервіс `orderReserveService`.
3. Інтеграція в роути в trim-style: 3 точки — POST /, item PATCH status canceled, order cancel.
4. Тестовий сценарій: створити замовлення на warehouse-позицію → подивитися `v_product_stock` → побачити, що qty_available впала.

**Критерий готовности:**
- Створення замовлення на warehouse-item → `inventory_reserves` має запис з status='active' і `reserve_id` записаний в item.
- Скасування замовлення → status='canceled' для всіх його резервів.
- TS зелений.

**Тест проверки:**
- Створити замовлення на 5 шт товару, у якого `qty_on_hand=10` → `qty_available=5`.
- Скасувати замовлення → `qty_available=10`.
- Створити друге замовлення на 7 шт коли є вже 5 в резерві → має повернути 422 `INSUFFICIENT_AVAILABLE`.
- Замовлення з `source_type='supplier'` не створює резерву.

**Риск:** середній — додає сценарій відмови при створенні замовлення (`INSUFFICIENT_AVAILABLE`); потенційно лякає менеджерів.
**Откат:** rollback міграції (drop column + drop RPCs); endpoint поведінка повертається до старої.
**Оценка сложности:** M
**Оценка количества прогонов:** 1
**Решения этапа:** менеджер не може продати товар, який уже обіцяний клієнту, але кашир — ще може (Етап 3).

# ЭТАП 3 — process_sale враховує резерви (P0)

**Цель:** касир не може продати товар, який зарезервований замовленням.

**Причина порядка:** після того як резерви існують і реально створюються (Етап 2), POS має на них дивитись. Раніше — нема сенсу.

**Зависимости:** Етапи 1 та 2.

**Использовать существующее:**
- RPC `process_sale` (поточна, з міграції 031).
- VIEW `v_product_stock` (Етап 1).

**Создать:**
- Міграція з оновленою версією `process_sale`: замість `SELECT qty_on_hand FROM products … FOR UPDATE` — `SELECT qty_available FROM v_product_stock … FOR UPDATE` (потребує функціональної заміни — fetch з products + sum з inventory_reserves у тій же транзакції з blockuваням).
- Альтернативно: окрема helper-функція `get_available_locked(product_id)`.

**Изменить:**
- БД: переписати тіло `process_sale` — додати перевірку `qty_available >= qty` замість `qty_on_hand >= qty`.
- `apps/web/src/features/pos/SearchPanel.tsx` — поряд із залишком показувати "(N в резерві)" якщо є.

**НЕ менять:**
- API касира (контракт ендпоінта не змінюється).
- Customer orders flow.
- Бонусну логіку.

**Поток работы:**
1. Написати нову версію `process_sale` як міграцію, але з новою назвою функції `process_sale_v2` — старий не чіпати, дати можливість відкоту.
2. Перемкнути виклик у `saleService.ts.createSale` з `db.rpc('process_sale', …)` на `'process_sale_v2'`.
3. Інтеграційний тест: створити замовлення з warehouse-резервом, спробувати продати той самий товар у POS на більшу кількість ніж available — має повернути `INSUFFICIENT_STOCK`.
4. Якщо все OK — наступним релізом видалити старий `process_sale`.

**Критерий готовности:**
- `INSUFFICIENT_STOCK` повертається коли qty_available недостатньо, навіть якщо qty_on_hand достатньо.
- Звичайні продажі без резервів працюють так само як раніше.

**Тест проверки:**
- 10 шт на складі, 3 у резерві замовлення → касир продає 8 шт → відмова.
- 10 на складі, 0 у резерві → касир продає 8 → успіх.
- Касир продає 5 шт замовлення A, потім скасовує — qty_available повертається.

**Риск:** високий — серце POS. Тому **версіонуванням `_v2` + feature flag через env `USE_RESERVE_AWARE_SALE`**.
**Откат:** змінити константу env та перезапустити сервер. БД-зміни inert (стара функція жива).
**Оценка сложности:** S (зміна одного RPC + один виклик)
**Оценка количества прогонов:** 1
**Решения этапа:** ціла система перестає продавати "повітря".

**ВЫПОЛНЯТЬ ОТДЕЛЬНЫМ ПРОГОНОМ МОДЕЛИ**

# ЭТАП 4 — Атомарне завершення замовлення в одну транзакцію (P0)

**Цель:** при `POST /customer-orders/:id/complete` мають атомарно статися: створення `sale`, декремент `qty_on_hand`, списання резерву (`status='consumed'`), фіскалізація, зарахування передоплати як кредит.

**Причина порядка:** Етапи 2-3 дали резерв, але "видача" клієнту — це інший флоу, який зараз ламає інваріант. Без цього резерв висить вічно або, ще гірше, споживається без фіскалізації.

**Зависимости:** Етапи 1, 2, 3.

**Использовать существующее:**
- RPC `process_sale_v2`.
- RPC `consume_order_reserve`.
- Endpoint `POST /:id/complete` (повністю замінюється).

**Создать:**
- БД: RPC `complete_customer_order(p_order_id, p_cashier_id, p_shift_id, p_payment_method, p_cash_amount, p_card_amount, p_is_fiscal)` — всередині:
  - читає всі items зі статусами `arrived`,
  - формує `p_items` JSONB і викликає `process_sale_v2` (inline як вкладену функцію, або повторює логіку),
  - оновлює resp. резерви на `consumed` для warehouse-items,
  - переносить prepayment у новий sale як `cash_amount` (зараховує борг),
  - проставляє `customer_orders.status='completed'`, items.item_status='handed',
  - повертає `{ sale_id, sale_number }`.

**Изменить:**
- `server/src/routes/customerOrders.ts` — `POST /:id/complete` тонкий wrapper навколо нового RPC. Видалити поточну "перевірку оплати" (тепер це робить RPC).

**НЕ менять:**
- POS code path для звичайного продажу.
- Створення замовлень (Етап 2 покрив).
- UI кнопки "Видати" — поки що залишається тут же.

**Поток работы:**
1. RPC `complete_customer_order` з повною логікою.
2. Оновити роут.
3. Тест: створити замовлення з warehouse-item, передоплатою → ready → complete → перевірити sale створено, qty_on_hand впав, резерв consumed, order completed.

**Критерий готовности:**
- `customer_orders` після complete має статус `completed`, посилання на новий `sale_id` (додати колонку).
- `products.qty_on_hand` знизився на qty виданого.
- `inventory_reserves.status='consumed'` для відповідних items.
- `sales` має запис з prepayment як cash payment.

**Тест проверки:**
- Замовлення на 3 warehouse-позиції з передоплатою 1000 → видати → у `sales` запис, у `inventory_reserves` всі три consumed, передоплата врахована.
- Замовлення лише на supplier-items (без резервів) → видати → sale є, резервів не торкається.
- Якщо передоплата перевищує суму → повернути overpay як cash_out у cash_operation.

**Риск:** високий — це бізнес-критичний шлях. Потрібен dry-run на staging БД.
**Откат:** старий endpoint лишити закоментованим; нова кнопка в UI має флаг env `USE_ATOMIC_COMPLETION`.
**Оценка сложности:** L (велика RPC, багато крайових випадків)
**Оценка количества прогонов:** 1
**Решения этапа:** жодна "видача" не залишає неконсистентний стан. Кожне виданне замовлення — фіскальний документ з реальним декрементом стоку.

**ВЫПОЛНЯТЬ ОТДЕЛЬНЫМ ПРОГОНОМ МОДЕЛИ**

# ЭТАП 5 — Готові замовлення в POS-у касира (P1)

**Цель:** касир бачить у касовому інтерфейсі "Готові замовлення" → кліком підвантажує items + клієнта + передоплату → одним натиском видає.

**Причина порядка:** Етап 4 дав API. Тепер UI, щоб ним зручно користатися.

**Зависимости:** Етап 4.

**Использовать существующее:**
- `apps/web/src/features/pos/POSPage.tsx` (поточний касовий екран).
- Stores: `posStore.ts`.
- Endpoint `GET /api/v1/customer-orders?status=ready`.

**Создать:**
- Компонент `ReadyOrdersPanel.tsx` усередині POS — sidebar/dropdown зі списком готових замовлень.
- Метод `loadReadyOrderToCart(orderId)` у posStore — підвантажує items, customer, prepayment_as_credit.
- Кнопка "Видати з замовлення" → виклик `complete_customer_order` (Етап 4).

**Изменить:**
- `apps/web/src/features/pos/POSPage.tsx` — додати точку входу для готових замовлень.
- `apps/web/src/stores/posStore.ts` — додати state `loadedOrderId: string | null` та логіку "ця корзина прийшла з замовлення".
- `apps/web/src/features/pos/PaymentModal.tsx` — якщо `loadedOrderId` встановлено, при підтвердженні викликати `complete_customer_order` замість `process_sale`.

**НЕ менять:**
- Звичайний касовий флоу без замовлення.
- Сторінку Замовлень (просто проставляє статус "ready").

**Поток работы:**
1. UI-панель у POS.
2. Логіка завантаження кошика з замовлення.
3. Розгалуження в PaymentModal на основі `loadedOrderId`.
4. Smoke: створити замовлення → довести до ready → у POS вибрати → видати → перевірити sale + статус completed.

**Критерий готовности:**
- На POS видно бейдж із кількістю готових замовлень.
- Завантаження замовлення підставляє items + клієнта.
- Підтвердження оплати створює правильний sale.

**Тест проверки:**
- Підвантажене замовлення з передоплатою — у вікні оплати видно "залишок Х".
- Касир не може редагувати позиції з підвантаженого замовлення (захист від дрейфу).
- Скасувати завантаження → корзина чиститься.

**Риск:** середній — змінює знайомий касовий екран; потрібне навчання касирів.
**Откат:** прибрати кнопку "Готові замовлення" з UI; стара кнопка "Видати" в Замовленнях лишається.
**Оценка сложности:** L
**Оценка количества прогонов:** 1-2 (UI + інтеграція як 2 прогони якщо складно)
**Решения этапа:** касир завершує замовлення з тим самим звичним інтерфейсом, що й звичайний продаж. Жодних окремих модалок з невідомими кнопками.

# ЭТАП 6 — Складська комірка / picking (P1)

**Цель:** після прибуття всіх позицій комірник присвоює замовленню номер комірки. Касир бачить його в POS і клієнт — у Telegram-сповіщенні.

**Причина порядка:** не блокує Етап 5, але різко зменшує час обслуговування. Природне доповнення.

**Зависимости:** Етап 4 (для атомарного завершення); сповіщення вже є в `saleService.markReadyForPickup`.

**Использовать существующее:**
- `sales.pickup_cell` (вже існує, міграція 035).
- `customer_orders.pickup_deadline_at` (вже існує).
- `messengerService` (для Telegram сповіщень).

**Создать:**
- БД: `customer_orders.pickup_cell VARCHAR(20) NULL`.
- ендпоінт: `PATCH /customer-orders/:id/cell` — body `{ cell: string }`.
- сторінка `apps/web/src/features/orders/PickingPage.tsx` — список ready-замовлень з полем для введення комірки.

**Изменить:**
- `OrdersPage.tsx` (новий inbox) — у списку Готових до видачі показувати `pickup_cell`.
- POS `ReadyOrdersPanel.tsx` (Етап 5) — показувати комірку поруч з номером.
- Telegram-повідомлення "замовлення готове" — додати комірку.

**НЕ менять:**
- Решту флоу замовлень.
- Sales.pickup_cell (це інша таблиця для відкладених чеків).

**Поток работы:**
1. Міграція колонки.
2. PATCH endpoint + service.
3. PickingPage з простою таблицею.
4. Телеграм-повідомлення оновити.

**Критерий готовности:**
- Комірник на PickingPage позначає комірку → запис у `customer_orders.pickup_cell`.
- Клієнт у Telegram отримує "Замовлення готове. Комірка: А-12".
- Касир бачить комірку в POS-панелі.

**Тест проверки:**
- Замовлення без комірки → Telegram-повідомлення без рядка.
- Замовлення з коміркою → Telegram містить її.
- Зміна комірки → новий PATCH перезаписує (без історії).

**Риск:** низький — additive.
**Откат:** drop column.
**Оценка сложности:** M
**Оценка количества прогонов:** 1
**Решения этапа:** клієнт точно знає куди йти; касир не шукає товар у магазині.

# ЭТАП 7 — COGS-snapshot на sale_items (P2)

**Цель:** записувати ціну закупки в момент продажу. Без цього звіт "прибуток" завжди приблизний / переписується.

**Причина порядка:** prerequisite для Етапів 8 (комісії з прибутку, а не з виручки) і для коректних звітів. Незалежний від 1-6 за технікою, але робиться після — щоб не плутати ризики.

**Зависимости:** жодних структурних.

**Использовать существующее:**
- `sale_items` таблиця.
- `process_sale_v2` (Етап 3).
- `products.buy_price` як джерело.

**Создать:**
- БД: `sale_items.cost_price INTEGER NULL` (default 0, бо backfill старих неможливий).
- Звіт `GET /api/v1/reports/profit?from=&to=` — повертає виручку, собівартість, маржу.

**Изменить:**
- `process_sale_v2` — при insert у `sale_items` записувати `cost_price = products.buy_price` (зчитано в той самий FOR UPDATE).
- `server/src/services/reportService.ts` — додати методи profit-звіту.
- `apps/web/src/features/reports/DailyReport.tsx` — рядок "Прибуток за період".

**НЕ менять:**
- POS поведінку (UI).
- Структуру звітів окрім нової секції.
- Старі sales (backfill = NULL, фільтр у звіті).

**Поток работы:**
1. Міграція + оновлення RPC.
2. Звіт endpoint + UI рядок.
3. Тест: продати товар, у БД `sale_items.cost_price` дорівнює buy_price; звіт відображає маржу.

**Критерий готовности:**
- Нові продажі мають `cost_price > 0` (якщо у товару є buy_price).
- Звіт повертає валідну суму прибутку для періодів після релізу.

**Тест проверки:**
- Продажа: cost_price == buy_price у момент продажу.
- Зміна buy_price ПІСЛЯ продажу не міняє cost_price у sale_items (snapshot).
- Звіт за діапазон, який охоплює і старі (NULL), і нові продажі — старі вилучаються або позначаються `unknown_cost`.

**Риск:** низький (нова колонка, не блокуюча).
**Откат:** drop column; RPC залишається працездатним з NULL.
**Оценка сложности:** M
**Оценка количества прогонов:** 1
**Решения этапа:** реальний прибуток замість виручки — основа для всіх грошових рішень.

**ВЫПОЛНЯТЬ ОТДЕЛЬНЫМ ПРОГОНОМ МОДЕЛИ**

# ЭТАП 8 — Комісії менеджерам (P2)

**Цель:** кожен менеджер бачить, скільки заробив за період; власник бачить, скільки винен.

**Причина порядка:** залежить від COGS (Етап 7) — комісія має нараховуватися з маржі, а не з виручки. Інакше дешеві товари «розбагатять» менеджера за рахунок власника.

**Зависимости:** Етап 7.

**Использовать существующее:**
- `sales.manager_id` (вже є).
- `salary_payments` таблиця.
- `shop_settings` для ставок.

**Создать:**
- БД: `shop_settings.commission_pct INTEGER NULL` (% від маржі, default 0).
- Можна — окрема таблиця `staff_commission_rules(user_id, pct)` якщо ставки персональні.
- ендпоінт: `GET /api/v1/salary/preview?from=&to=&user_id?` — повертає `{ user_id, sales_count, revenue, margin, commission, already_paid }`.
- ендпоінт: `POST /api/v1/salary/payouts` — створює `salary_payments` запис.
- сторінка `StaffSalaryPage.tsx` (вже існує — розширити).

**Изменить:**
- `apps/web/src/features/staff/StaffSalaryPage.tsx` — додати блок "Комісія за період" з preview і кнопкою "Виплатити".

**НЕ менять:**
- Логіку продажу — комісія розраховується на читанні, не на запису.
- Loyalty/бонуси (не пов'язано).

**Поток работы:**
1. Міграція налаштувань ставки.
2. Service-метод `calculateCommissionForPeriod`.
3. Endpoints.
4. UI preview + payout.

**Критерий готовности:**
- Власник вибирає період + менеджера → бачить нараховану комісію.
- Виплата записує `salary_payments` + `cash_operations` (type='out', if cash).

**Тест проверки:**
- Менеджер без жодних продажів — комісія 0.
- Менеджер з продажем на 1000 грн (маржа 200, ставка 10%) — комісія 20.
- Повторна виплата за той самий період — попередження, не дубль.

**Риск:** середній — грошові операції з співробітниками. Має бути audit.
**Откат:** drop column settings.commission_pct; UI блок прибрати.
**Оценка сложности:** M
**Оценка количества прогонов:** 1
**Решения этапа:** прозора система мотивації, автоматизовані виплати.

**ВЫПОЛНЯТЬ ОТДЕЛЬНЫМ ПРОГОНОМ МОДЕЛИ**

# ЭТАП 9 — Черга фонових задач (P0 інфра, P2 в часі)

**Цель:** замінити поточний `setInterval(processOrderDeadlines, 6h)` на реальну чергу з retry, idempotency, видимістю стану.

**Причина порядка:** не блокує бізнес-функції 1-8, тому виноситься нижче. Але без нього кожна нова фоновa задача (нагадування про оплату, синхронізація з постачальниками, ребілд індексів) — це новий setInterval, який ламається при перезапуску.

**Зависимости:** жодних бізнесових.

**Использовать существующее:**
- `processOrderDeadlines` (мігрувати в чергу).
- Postgres (як storage для черги).

**Создать:**
- БД: таблиця `jobs (id UUID, type VARCHAR, payload JSONB, status VARCHAR DEFAULT 'pending', attempts INT DEFAULT 0, run_at TIMESTAMPTZ, locked_until TIMESTAMPTZ NULL, result JSONB NULL, created_at, updated_at)`.
- БД: індекс `(status, run_at) WHERE status='pending'`.
- сервіс: `server/src/services/jobQueue.ts` — `enqueue(type, payload, runAt?)`, `worker tick()` з `FOR UPDATE SKIP LOCKED`.
- handler registry: `jobs/handlers/orderDeadline.ts`.
- ендпоінт: `GET /api/v1/admin/jobs?status=` — debug-перегляд для власника.

**Изменить:**
- `server/src/index.ts` — `setInterval(workerTick, 30s)` замість прямого виклику `processOrderDeadlines`.
- `processOrderDeadlines` — переписати як enqueuer (1 раз на 6 год кладе `{ type: 'check_overdue_orders' }`).

**НЕ менять:**
- Бізнес-логіку нагадувань (Telegram-повідомлення).
- Інші сервіси.

**Поток работы:**
1. Міграція + сервіс.
2. Перенесення наявного worker.
3. UI-сторінка debug-перегляду (за бажанням, не блокує MVP).

**Критерий готовности:**
- Перезапуск сервера: pending jobs дочекалися worker'а й виконалися.
- Якщо handler впав → `attempts++`, retry з exponential backoff.
- 3+ невдачі → status='failed', не повторюємо.

**Тест проверки:**
- Інстансу 1 і 2 одночасно — кожен job виконується **рівно один раз** (через FOR UPDATE SKIP LOCKED).
- Job з `run_at` у майбутньому не береться раніше.
- Падіння → retry до 3 разів.

**Риск:** середній. На staging — обовʼязково.
**Откат:** залишити стару setInterval-логіку; jobs table inert.
**Оценка сложности:** L
**Оценка количества прогонов:** 1-2
**Решения этапа:** фундамент для всіх майбутніх async-фіч (синхронізація з постачальниками, SMS-сповіщення, тощо).

**ВЫПОЛНЯТЬ ОТДЕЛЬНЫМ ПРОГОНОМ МОДЕЛИ**

## СТОП-ПРАВИЛА

1. **Не починати наступний етап до підтвердженого «зеленого» попереднього.** «Зелений» = міграція застосована, TS чистий, smoke-test з критерію готовності пройдений на staging-БД.

2. **Етапи 3, 4 та 9 — деплоїти через feature flag** (env-перемикач), не «гарячий» викочуванням. Сервер запускається з вимкненим прапором, перевіряється, потім вмикається.

3. **Перший деплой DDL — у нерабочий час магазину.** Особливо Етапи 2, 4, 7 (зачіпають серце транзакцій).

4. **Релізний порядок завжди:** server → міграція → frontend. Старий фронтенд має продовжувати працювати з новим бекендом ≥24 год.

5. **Кожен етап має зворотний шлях** (Откат). Якщо `process_sale_v2` ламається — змінюємо env-флаг назад на `process_sale`. Якщо `complete_customer_order` ламається — повертаємось до старої логіки в роуті.

6. **Етапи 5, 6, 8, 10 — суто адитивні UI/функції.** Можна виключити з релізу через приховування кнопок у Sidebar, без міграційного відкату.

7. **Параметри одного етапу — НЕ використовувати у двох прогонах.** Якщо обсяг етапу не вкладається в одну сесію моделі, **розщепити на 5.a / 5.b** перед запуском, а не «дописати в наступному прогоні».

8. **Жоден етап не має чіпати «не свій» файл.** Якщо план говорить «не міняти POSPage» — будь-яка зміна там у тому ж прогоні = негативний результат прогону, навіть якщо тест проходить.
