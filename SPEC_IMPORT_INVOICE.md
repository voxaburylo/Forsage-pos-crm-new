# SPEC: Імпорт накладної (розумний парсер)

## Концепція

Оператор отримує список товарів від постачальника у будь-якому форматі
(Excel, CSV, скопійований текст з сайту) і вставляє його в систему.
Система сама розбирає, знаходить товари в базі, пропонує варіанти для
невідомих і формує накладну на приймання.

---

## Таблиці БД

```sql
-- Сесія імпорту (лог кожного імпорту)
CREATE TABLE import_sessions (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id         UUID NOT NULL REFERENCES tenants(id),
  supplier_id       UUID REFERENCES suppliers(id),
  source_type       TEXT NOT NULL, -- 'excel', 'csv', 'clipboard_text', 'manual'
  raw_content       TEXT,          -- оригінальний вставлений текст (для дебагу)
  total_rows        INT DEFAULT 0,
  matched_rows      INT DEFAULT 0, -- знайдено точно
  fuzzy_rows        INT DEFAULT 0, -- знайдено схоже
  new_rows          INT DEFAULT 0, -- нові товари
  status            TEXT NOT NULL DEFAULT 'draft',
  -- draft / confirmed / cancelled
  receipt_id        UUID REFERENCES inventory_receipts(id), -- після підтвердження
  created_by        UUID REFERENCES users(id),
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at        TIMESTAMPTZ
);

-- Рядки імпорту (кожна позиція зі списку)
CREATE TABLE import_session_rows (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id             UUID NOT NULL REFERENCES tenants(id),
  import_session_id     UUID NOT NULL REFERENCES import_sessions(id),
  row_number            INT NOT NULL,
  raw_article           TEXT,          -- артикул як є у файлі
  raw_name              TEXT,          -- назва як є у файлі
  raw_quantity          TEXT,          -- кількість як є
  raw_price             TEXT,          -- ціна як є
  normalized_article    TEXT,          -- нормалізований артикул
  quantity              NUMERIC(12,3),
  price_kopecks         INTEGER,
  match_type            TEXT,          -- 'exact' / 'fuzzy' / 'new' / 'skipped'
  matched_product_id    UUID REFERENCES products(id),
  confidence_score      NUMERIC(4,2),  -- 0-100, наскільки впевнені в збігу
  user_decision         TEXT,          -- 'accepted' / 'rejected' / 'create_new'
  new_product_name      TEXT,          -- назва для нового товару
  created_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## Алгоритм парсера

### Крок 1 — Визначення формату вхідних даних

```
IF файл .xlsx або .xls:
  → Читати через SheetJS
  → Перша рядок = заголовки (або автовизначення)

IF файл .csv:
  → Читати через PapaParse
  → Автовизначення роздільника (, або ; або \t)

IF текст із буфера (clipboard):
  → Спробувати розбити по \t (таблиця скопійована з Excel/сайту)
  → Якщо не виходить → розбити по \n і аналізувати кожен рядок
  → Regexp витягує числа (кількість, ціна) і текст (артикул, назва)
```

### Крок 2 — Автовизначення колонок

```
Система аналізує заголовки і визначає колонки:
  Артикул/Код/Article/Арт     → article
  Назва/Найменування/Name     → name
  Кількість/Qty/Кіл-ть/Кол   → quantity
  Ціна/Price/Вартість         → price

Якщо заголовків немає — евристика:
  - Колонка де всі значення схожі на артикули (букви+цифри+дефіс) → article
  - Колонка де числа без дробових → quantity
  - Колонка де числа з дробовими або великі числа → price
  - Найдовша текстова колонка → name
```

### Крок 3 — Нормалізація артикулів

```javascript
function normalizeArticle(raw) {
  return raw
    .toUpperCase()
    .replace(/[\s\-_\.]/g, '')  // прибрати пробіли, дефіси, підкреслення, крапки
    .replace(/^0+/, '')          // прибрати ведучі нулі
    .trim()
}

// Приклади:
// "BP-1234"  → "BP1234"
// "bp 1234"  → "BP1234"
// "BP_1234"  → "BP1234"
// "0BP1234"  → "BP1234"
// Всі чотири знайдуть один і той самий товар
```

### Крок 4 — Пошук товару в базі

```
Для кожного рядка:

1. ТОЧНИЙ ЗБІГ (confidence: 100):
   normalize(row.article) = normalize(product.article)
   АБО normalize(row.article) = normalize(barcode.barcode)
   АБО normalize(row.article) = normalize(product_supplier_codes.code)
   → match_type = 'exact'

2. НЕЧІТКИЙ ЗБІГ (confidence: 60-95):
   a) normalize(row.article) схожий на normalize(product.article)
      (Levenshtein distance <= 2 символи)
   b) АБО назва товару збігається на 80%+
      (порівняння через trigram similarity)
   → match_type = 'fuzzy', показати варіанти для вибору

3. НЕ ЗНАЙДЕНО (confidence: 0):
   → match_type = 'new'
   → форма швидкого створення з заповненими полями з файлу
```

### Крок 5 — Результат парсингу

```
Система групує рядки за категоріями:
  🟢 Знайдено точно (N позицій) — авто в накладну
  🟡 Знайдено схоже (N позицій) — потрібне підтвердження
  🔴 Новий товар (N позицій)    — потрібне створення

Показує таблицю з трьома секціями.
Оператор підтверджує жовті і заповнює червоні.
```

## API ендпоінти

```
POST /api/import/parse
  body: { supplier_id, source_type, content: "raw text or base64 file" }
  → { session_id, rows: [...], summary: { exact, fuzzy, new } }

GET  /api/import/:sessionId
  → { session, rows }

PUT  /api/import/:sessionId/rows/:rowId
  body: { user_decision, matched_product_id?, new_product_name? }
  → оновлений рядок

POST /api/import/:sessionId/confirm
  → створює inventory_receipt + всі позиції
  → повертає { receipt_id }

DEL  /api/import/:sessionId
  → скасувати сесію імпорту (статус cancelled)
```

## Обробка помилок парсингу

```
Файл не читається:
  → "Не вдалось прочитати файл. Спробуйте зберегти як .csv і повторити"

Не знайдено колонок:
  → Показати ручний маппінг: "Яка колонка містить артикул? [▼]"

Порожній результат:
  → "Знайдено 0 рядків. Перевірте формат файлу або вставте текст вручну"

Занадто багато нових (>80%):
  → Попередження: "78% позицій не знайдено. Можливо вибраний не той постачальник?"
```

## Налаштування маппінгу для постачальника

Зберігати маппінг колонок для кожного постачальника, щоб не визначати щоразу:

```sql
ALTER TABLE suppliers
  ADD COLUMN import_column_map JSONB;
-- Приклад:
-- { "article": "Код", "name": "Найменування", "qty": "Кількість", "price": "Ціна" }
```

Після першого успішного імпорту — запитати: "Зберегти маппінг для цього постачальника?"
